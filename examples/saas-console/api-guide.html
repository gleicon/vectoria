API Guide — Control Plane
All requests require Bearer token auth. Two key types exist: the admin key (full access) and per-tenant vtk_ keys (scoped to that tenant's indexes only). Code examples use your connected server URL automatically.

Authentication & access model

Every request must include Authorization: Bearer <key>. The server infers the caller identity from the key — no separate login endpoint.

Tenant keys are prefixed vtk_ and are generated by the server. They only resolve indexes within that tenant's namespace ({tenant}/{index} internally); requests for other tenants' indexes return 404, not 403, to prevent namespace enumeration.

Tenant management admin key only

Tenants are isolated namespaces. Creating a tenant generates a vtk_ API key. The tenant creates its own indexes self-service via POST /indexes using that key.

List tenants

GET /admin/tenants

Create tenant

POST /admin/tenants

Returns the generated api_key — save it, it cannot be retrieved later (only rotated).

Delete tenant

DELETE /admin/tenants/{name}

Cascade-deletes all indexes belonging to the tenant. Irreversible.

Rotate API key

POST /admin/tenants/{name}/rotate-key

Invalidates the current vtk_ key immediately and returns a new one.

List a tenant's indexes

GET /admin/tenants/{name}/indexes

Index management tenant key admin key

Indexes are the unit of product storage and search. Each tenant manages its own indexes with its vtk_ key. The admin key manages system indexes (flat namespace). Tenant indexes live in a separate namespace and are not visible to admin via GET /indexes.

URL name vs. internal key: /indexes/catalog with a food tenant key resolves internally to food/catalog. The same URL with the admin key resolves to a flat system index catalog. Always use the correct key for the target namespace.

List indexes

GET /indexes

Admin sees system indexes only. Tenant sees its own indexes only.

Create index

POST /indexes

Index names: lowercase letters, digits, hyphens, underscores. Max 64 chars.

Delete index

DELETE /indexes/{name}

Data ingestion tenant key

Products are indexed into a named index. The tenant key + index name in the URL determine where data goes. Always create the index first.

Index a product

POST /indexes/{name}/products

id is your product ID (string). text is the searchable content. metadata holds arbitrary JSON returned in search results.

Rebuild search index (reindex)

POST /indexes/{name}/admin/reindex

BM25 text index is in-memory — lost on server restart. Call this after restart to rebuild from stored products. Runs in background; returns immediately.

Search tenant key

Search products

POST /indexes/{name}/search

mode: hybrid (default), bm25 (keyword), semantic (vector). limit: max results (default 15).

Index stats

GET /indexes/{name}/admin/stats

Search overrides tenant key

Overrides let you boost, inject, or suppress products for specific queries. All override routes follow the same pattern under /indexes/{name}/admin/.
